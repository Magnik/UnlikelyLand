# UnlikelyLand

A persistent, text-driven, weird-comedy multiplayer RPG for the browser (and, later, an installable PWA / Android app). You wash up on a strange island, spend regenerating stamina on absurd expeditions, make choices that quietly shape who your character becomes, fight the occasional militant recliner, sometimes die, and slowly work toward escaping.

The defining rule: **the server is authoritative and the AI only ever proposes.** Encounters can be generated by a local LLM, but every reward, stat change, combat result, item, and death is decided by server-side game rules. The game is fully playable with the AI switched off.

> New here as a developer? Start with [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md).

## Features

- **Expeditions** in seven flavours — Explore, Fight, Scavenge, Socialize, Investigate, Train, Work — each with its own encounter pool, stat-check tendencies, reward bias, and drop-slot bias.
- **AI or fallback encounters.** A local LLM proposes `encounter.v1` narrative + choices; the server validates, moderates, logs, and resolves. With AI off, 130+ hand-curated fallback encounters across seven pools keep the game fully playable — rating-safe and biased by the player's preferences.
- **Inventory & equipment.** Server-authoritative inventory with six slots (weapon, armor, tool, trinket, consumable, companion) and six rarities (common → absurd). Equip/unequip (one item per non-consumable slot), use consumables, and a single centralized **effective-stat** service (base + equipped modifiers) that combat and checks actually use.
- **Approved item drops.** Drops come only from the approved catalog via a centralized loot service (expedition-type, level, and rarity-weighted), are awarded inside the resolution transaction, logged to the reward/economy audit, and can never be granted twice for the same encounter.
- **AI item-concept review.** The AI may *propose* an item concept (name/description/slot/rarity only — never stats). A central validator moderates the text (family floor), generates a balanced, budget-capped stat block, and either auto-approves low-power common/uncommon that pass every rule or queues everything else for an admin.
- **Profile & settings.** Editable bio (server-moderated), public profile (public-only fields), content rating (Family / PG-13 / R, default PG-13), and **structured** story-style preferences (a closed set of toggles — no free-form prompt injection) that steer AI generation and bias fallback selection.
- **Progression feedback.** XP/level-up, currency, reputation, crafting, item drops, equipment changes, stat nudges, and death/revival are all surfaced after each encounter.
- **Economy.** Clams/Escape Tokens/Oddments/Notoriety, a player-to-player market (escrowed listings), and prestige/escape.
- **A shared, moderated world (Milestone 3).** Global **chat** (moderated, rate-limited, duplicate-protected, paginated, with guild tags and one-tap reporting); **friends + blocking** through one centralized relationship service (blocks are bidirectional and enforced across chat, mail, search and profiles); **private mail** (inbox/outbox, addressed by id or unique name); **public profiles** at `/u/[id]` with optional titles, achievements, equipment, recent activity, and in-context friend/block/report controls; **public achievements** (central evaluator, idempotent) + a public **activity feed**; **leaderboards** (level/wealth/reputation/victories/achievements — paginated, with your own rank, normal currency only); and **guilds** with tags, moderated names, and owner/officer/member roles (promote/demote/kick/transfer). Private Story Memory, AI logs, hidden stats, economy history and others' messages are **never** exposed to players.
- **Moderation & safety.** Players report messages/profiles; a role-gated **moderator console** (`/moderation`) reviews reports and hides/deletes/restores messages, mutes and warns; **admins** additionally ban/unban, grant/revoke moderator, and disband guilds. Every privileged action is written to an append-only **audit trail**, and a banned or demoted account's token stops working on the next request.
- **Admin tools.** AI on/off + force-fallback toggles, AI generation log, player lookup, character inventory + Story-Memory inspection, economy/reward log, item catalog browser, and item-concept review. All admin routes are role-gated; the moderation surface is separated so moderators never reach admin-only secrets.

### Game balance & content

All gameplay numbers live in [`apps/api/src/engine/rules.ts`](apps/api/src/engine/rules.ts) (rewards, loot weighting, item power budgets, stat-mod caps). Fallback encounters are modular JSON under [`apps/api/src/content/fallback/`](apps/api/src/content/fallback/), assembled by `index.ts` — add a file and list it there; no giant file to edit. New content can be authored and bulk-validated with `node scripts/import-fallback-content.mjs <authoring.json>`.

## Stack

| Layer | Choice |
|---|---|
| Frontend | Next.js 14 (App Router) + React, mobile-first, PWA manifest |
| Backend | NestJS + TypeScript |
| Database | PostgreSQL + Prisma ORM |
| Validation | Zod (shared `@unlikelyland/contracts` package) |
| Auth | Username/password, JWT (bcrypt hashing) |
| AI | Ollama-compatible provider behind a swappable gateway |
| Realtime/jobs | Redis (reserved for chat + BullMQ in later milestones) |
| Deploy | Docker Compose + Caddy (automatic HTTPS) |

Monorepo layout:

```
packages/contracts   # shared zod schemas + types (enums, encounter.v1, API DTOs)
apps/api             # NestJS game server (engine, AI gateway, REST API)
apps/web             # Next.js client (the whole UI)
docs/                # architecture, game design, deployment, roadmap, Android notes
deploy/, scripts/    # Caddy config, Postgres backup/restore
```

## Prerequisites

- Node.js 20+
- Docker + Docker Compose (for the containerised path)
- (Optional) [Ollama](https://ollama.com) for AI-generated encounters — the game works without it.

## Quick start — Docker (recommended)

```bash
cp .env.example .env
# Edit .env: set strong POSTGRES_PASSWORD, JWT_SECRET, ADMIN_PASSWORD.
docker compose up --build
```

- Web app: <http://localhost:3000>
- API: <http://localhost:4000> (health at `/health`)

The API container runs migrations and seeds the world (region sets, item catalog, achievements, AI settings, and the admin account) automatically on first start. Register a player at `/register`, or log in as the seeded admin (`ADMIN_USERNAME` / `ADMIN_PASSWORD`) to reach `/admin`.

## Quick start — local (no Docker)

You need a PostgreSQL instance. Point `DATABASE_URL` at it (Prisma reads `.env` from `apps/api/`, so copy the root `.env` there too):

```bash
cp .env.example .env
cp .env apps/api/.env        # Prisma CLI reads env from the workspace dir
npm install
npm run build:contracts      # build the shared package first

# Database
npm run prisma:migrate -w @unlikelyland/api   # create/apply migrations
npm run prisma:seed -w @unlikelyland/api      # seed the world + admin

# Run the two apps (separate terminals)
npm run dev:api              # NestJS on :4000
npm run dev:web              # Next.js on :3000
```

## AI (Ollama) — optional

The AI gateway talks to any Ollama-compatible `/api/chat` endpoint.

1. Install Ollama and pull a model: `ollama pull llama3.1:8b`
2. In `.env` set `OLLAMA_BASE_URL` and `OLLAMA_MODEL`.
   - Running the API in Docker but Ollama on your host? Use `http://host.docker.internal:11434` (already wired via `extra_hosts`).
3. AI can be toggled at runtime from the Admin dashboard (Enable/Disable AI, Force fallback). Every generation attempt is validated against `encounter.v1`, moderated, and logged; on any failure or timeout the server falls back to seeded content. **The game never breaks when the AI is offline.**

## Useful scripts

```bash
npm run test          # contracts + API engine/moderation/content/loot/validator tests (vitest)
npm run typecheck     # typecheck every workspace
npm run build         # build contracts → api → web

# Database (run inside apps/api, or with -w @unlikelyland/api)
npm run prisma:generate -w @unlikelyland/api   # regenerate the Prisma client
npm run prisma:migrate  -w @unlikelyland/api   # create + apply a dev migration
npm run prisma:deploy   -w @unlikelyland/api   # apply pending migrations (prod/CI)
npm run prisma:seed     -w @unlikelyland/api   # seed regions, 38-item catalog, achievements, admin
```

Migrations are additive and run automatically (`prisma migrate deploy`) on API container start; the seed is idempotent and safe to re-run. The latest migration, `0007_integrity_hardening`, adds real foreign keys (`ON DELETE CASCADE`) + reverse-lookup indexes to the social-graph join tables (`FriendRequest` / `Friendship` / `BlockedUser`), switches `Guild.owner` from `RESTRICT` to `CASCADE` (so deleting a founder disbands the guild instead of blocking the delete), indexes the previously-unindexed `MarketListing` / `ChatMessage` FK + channel columns, and adds a `(characterId, escapeCount)` unique on `EscapeRecord` that makes prestige escape a one-shot. It deletes any orphaned social rows before adding the FKs, so it is safe on existing data.

### Admin & moderator bootstrap

The first **admin** account is created by the seed from `ADMIN_USERNAME` / `ADMIN_PASSWORD` (one-time; idempotent). Log in and visit `/admin` (AI/economy/Story-Memory tools) and `/moderation` (the moderation console).

To appoint a **moderator**, an admin opens `/moderation`, searches the player by name, and clicks *Grant moderator* (this calls `POST /moderation/role`, which is admin-gated and audited). Additional admins are still provisioned out-of-band by setting `User.role = 'admin'` in the DB — the in-app role tool deliberately won't mint admins.

### How moderation, blocking, friends & guild permissions work

- **Roles.** `player` → `moderator` → `admin`. `RolesGuard` treats `admin` as a superset. The whole `/moderation` controller is `@Roles('moderator')`; its destructive handlers (ban, role changes, guild disband) are narrowed to `@Roles('admin')`. `/admin` (AI logs, raw Story Memory, economy ledger) stays admin-only, so moderators never see those secrets. **Role and ban state are re-read from the DB on every request**, so demoting or banning an account takes effect immediately, not when the JWT expires.
- **Reporting.** Any player can report a chat message, mail, or profile with a reason category (`POST /reports`). Reporters are never revealed; a player can't report the same message twice. Reports bump the target's `reportCount` so heavily-reported content surfaces first in the queue.
- **Chat moderation.** Moderators hide/delete (reversible soft-status `hidden`/`removed`) or restore messages, and mute/warn authors. Hidden/removed messages disappear from the public feed but stay in the DB; **every** action writes an append-only `ModerationAction` row (actor + target + reason + time), viewable at `/moderation` → audit.
- **Blocking is bidirectional** and centralized in `RelationshipService`. If either party blocks the other, neither sees the other's chat messages, neither can mail or friend-request the other, neither appears in the other's player search, and the blocked party gets a 404 (not a "you're blocked") when viewing the profile. Blocking also tears down any existing friendship + pending requests in one transaction.
- **Friends.** Search → send request → accept/decline/cancel/remove. You can't friend yourself, can't duplicate a friendship, and can't request someone in a block relationship with you. A mutual pending request auto-accepts.
- **Guild permissions.** `owner` (founder) can promote/demote officers, kick, transfer ownership, and edit the description/tag; `officer` can kick members; `member` can leave. The owner can't leave a populated guild without transferring ownership or removing everyone first. Names, tags, and descriptions are moderated; all multi-row changes are transactional.

### Realtime

Chat gets a near-realtime nudge over **Server-Sent Events** (`GET /chat/stream`, native NestJS `@Sse`, no extra dependency): when anyone posts, the stream emits a tiny channel-type "pulse" (no message body), and the client immediately re-fetches through the normal block/channel-filtered list endpoint — so all the authorization rules stay in one place. **Polling remains the fallback** if the stream drops, and mail / activity feed still poll. This is single-instance (the in-process pulse); a multi-replica deploy would back the pulse with Redis pub/sub (or swap SSE for a socket.io gateway with the Redis adapter) — the chat service keeps its business rules out of the transport so either upgrade is drop-in. No configuration is required.

Chat now has **global, region, and guild channels** (region chat is scoped to your region set, guild chat to your guild). Guilds have a **bank** (deposit/withdraw Oddments) and a derived **level**. Leaderboards can be filtered to **your region set**. Sessions use a **short-lived access token + refresh token**: the web client refreshes transparently on a 401, and logout/ban invalidates outstanding tokens via a per-user token version.

## Known limitations & next steps

- **Realtime** — chat pushes via SSE (with polling fallback); mail/market still poll. A socket.io + Redis gateway and BullMQ jobs are the multi-instance upgrade (Redis is wired in Compose).
- **Tests: 136 DB-free unit tests** (`npm run test`) — engine/validator/loot/fallback, the social/moderation service logic (relationship/block, achievement idempotency + evaluator, friends/blocking, guild role enforcement + bank/level, moderation hardening), and the economy-critical services added in the production-readiness pass: **market** (atomic claim / no double-buy, ownership, escrow), **economy** (race-safe conditional spend), **death revive** (per-method one-shot claim), **prestige escape** (level gate + P2002→conflict), and **encounter resolution** (idempotent replay, conflict, downed-character guard) — all with mocked Prisma. A **Postgres-backed integration scaffold** also exists (`npm run test:integration -w @unlikelyland/api` with `TEST_DATABASE_URL` set) — one real bidirectional-block test, the place to grow full request-path coverage; it skips when no test DB is configured.
- **Consumables** currently restore stamina only; the effect model (`ConsumableEffectType`) is built to extend.
- **AI-approved items** are family-floor moderated and stat-generated server-side; they enter the global catalog, so keep an eye on the admin item-concept queue.
- **Realtime is single-instance.** Chat uses an in-process SSE pulse; running >1 API replica would need the pulse backed by Redis pub/sub (or a socket.io + Redis-adapter gateway). Fine for the documented single-VPS deploy.
- **Leaderboard time *periods* (weekly/monthly) are not implemented.** Region filtering is; windowing cumulative stats (level/wealth) needs periodic snapshots, which is a deliberate follow-up.
- **The moderation blocklist is a blunt regex** (now leetspeak-normalized) and the real slur list is loaded from ops config, not source. It's paired with player reporting + a human review queue as the backstop; `moderateText` is a single swappable seam for a model/API later.
- **No token denylist.** Bans/role changes rely on `AuthGuard` re-reading role + `bannedAt` + `tokenVersion` from the DB each request (so they take effect immediately within the JWT lifetime). Per-token rotation/denylist is the upgrade path.
- **Web tokens live in `localStorage`.** Access + refresh tokens are JS-readable, so a successful XSS could exfiltrate them. This is mitigated by a strict app + API Content-Security-Policy and standard security headers (added in the audit pass), but the hardened design is to move the **refresh token to an httpOnly, Secure, SameSite cookie** and keep only an in-memory access token — a focused follow-up that needs live end-to-end testing of the Next proxy + cookie flow.
- **Login brute-force is throttled per-username and per-IP**, but the per-IP bucket is weak behind the Next.js proxy (it collapses client IPs to one upstream address). The per-username throttle (added in the audit pass) is the real defense; forwarding the true client IP end-to-end (Caddy → Next → API) is the remaining hardening.
- **A character can hold two active expeditions only under a precise concurrent double-submit** (the guard is app-level; Prisma can't express the partial-unique index that would enforce it at the DB without schema drift). Low impact — it only wastes the player's own stamina — and is the documented follow-up.
- **`MessageReport` target ids are polymorphic** (`targetMessageId` points at either a chat or mail row depending on `targetType`), so they have no DB foreign key; reports can dangle if the underlying message is hard-deleted. Reports are moderation records with low orphan-harm, so this is left as a noted follow-up.
- Recommended next milestone: **httpOnly refresh-token cookies**, WebSocket realtime (Redis pub/sub) for multi-instance, guild progression (bank/tasks/leveling), weekly/seasonal leaderboard periods, crafting (Oddments sink), and growing the Postgres integration suite into full request-path authorization coverage.

### Production deployment checklist

1. `cp .env.example .env` and set: a real `JWT_SECRET` (≥32 random chars), a strong `POSTGRES_PASSWORD`, a strong `ADMIN_PASSWORD`, and `DOMAIN`/`ACME_EMAIL`. The API refuses to boot in production with the example `JWT_SECRET` or the default `ADMIN_PASSWORD`.
2. Deploy with the prod overlay so only Caddy is public: `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build` (postgres/redis/api/web stay on the internal network). The Hostinger one-click path uses `docker-compose.hostinger.yml`.
3. On first boot the API container runs `prisma migrate deploy` (all migrations, including `0007_integrity_hardening`) then the idempotent seed, then starts — verify `GET /health` returns `ok`.
4. Confirm HTTPS resolves for `DOMAIN` (Caddy obtains the certificate automatically) and that `:4000`/`:5432`/`:6379` are **not** reachable from the public internet.
5. Set up Postgres backups (`scripts/backup-postgres.sh`) on a schedule and test a restore (`scripts/restore-postgres.sh`).
6. If enabling AI, point `OLLAMA_BASE_URL` at a reachable model and flip `AI_ENABLED=true`; otherwise the game serves instant seeded fallback content.
